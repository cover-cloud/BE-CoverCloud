# CoverCloud 프로젝트 - 기술별 심화 가이드 (면접 대비)

## 🎯 이 가이드의 목적
면접에서 깊이 있는 질문을 받았을 때 명확하고 전문적으로 답변하기 위한 기술별 심화 설명입니다.

## 1️⃣ MSA (Microservices Architecture)

### 1.1 MSA를 선택한 이유

#### 프로젝트에서의 필요성
```
CoverCloud는 다음 3가지 독립적인 도메인을 가지고 있습니다:
- 사용자 관리 (User Service)
- 커버곡 관리 (Cover Service)
- 음악 정보 관리 (Music Service)

각 서비스가 서로 다른 비즈니스 로직과 데이터 모델을 가지므로,
독립적인 팀이 개발하고 배포할 수 있는 구조가 필요했습니다.
```

#### MSA의 장점 (실제 적용 사례)
```
1️⃣ 독립적 배포 (Independent Deployment)
   - User Service 수정 시 다른 서비스를 재배포할 필요 없음
   - 배포 리스크 감소
   
2️⃣ 기술 선택의 자유 (Technology Flexibility)
   - Music Service는 Python으로 변경 가능
   - 각 팀이 최적의 기술 선택 가능
   
3️⃣ 확장성 (Scalability)
   - 트래픽 많은 Cover Service만 수평 확장
   - 리소스 효율성 향상
   
4️⃣ 장애 격리 (Fault Isolation)
   - Music Service 장애가 User Service에 영향 없음
   - 전체 시스템의 가용성 증대
```

### 1.2 MSA의 도전 과제 (CoverCloud에서의 해결책)

#### ❌ 문제 1: 서비스 간 데이터 일관성
```
상황: Cover Service에서 User Service의 사용자 정보가 필요함

❌ 잘못된 방법
- 사용자 정보를 Cover DB에 복제 → 데이터 불일치 위험

✅ 해결책
- Feign Client로 실시간 조회
- Redis 캐싱으로 성능 최적화

코드:
@FeignClient(name = "user-service", url = "http://localhost:8081")

    @GetMapping("/api/user/profiles/bulk")
    fun getUsersByIds(@RequestParam ids: List<Long>): List<UserProfileDto>
}

// 사용
val userProfiles = userClient.getUsersByIds(userIds)
```

#### ❌ 문제 2: 분산 트랜잭션
```
상황: 커버곡 생성 시 Music Service 호출 필요

❌ 일관성 문제
if (musicResult 실패) {
    Cover 저장됨, Music 저장 안됨 → 불일치
}

✅ 해결책 1: 보상 트랜잭션 (Saga Pattern)
@Transactional

✅ 해결책 2: 비동기 처리 (향후 도입)
- 메시지 큐(RabbitMQ/Kafka)로 "music-created" 이벤트 발행
- 각 서비스가 독립적으로 처리
```

## 2️⃣ JWT (JSON Web Token)

### 2.1 JWT 구조와 동작 원리

```
JWT = Header.Payload.Signature

1. Header (토큰 타입, 알고리즘)
{
  "alg": "HS512",
  "typ": "JWT"
}

2. Payload (클레임 정보)
{
  "userId": 123,
  "iat": 1234567890,      // 발급 시간
  "exp": 1234671490       // 만료 시간 (1시간 후)
}

3. Signature (검증용 서명)
HMACSHA512(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret_key
)
```

### 2.2 Access Token vs Refresh Token

```
┌─────────────────────────────────────────────────────┐
│                   로그인                             │
└───────────────────┬─────────────────────────────────┘
                    │
        ┌───────────▼───────────┐
        │ JWT 토큰 발급         │
        └───────────┬───────────┘
                    │
      ┌─────────────┴─────────────┐
      │                           │
      ▼                           ▼
Access Token              Refresh Token
(짧은 유효기간)            (긴 유효기간)
1시간 유효                 7일 유효

메모리에 저장              Redis에 저장

API 요청 시 사용          Access Token 갱신에만 사용
매번 검증                  서버에서 검증
```

### 2.3 CoverCloud의 JWT 구현

```kotlin
// JWT 토큰 생성
fun generateToken(userId: Long): String {
    val now = Date()
    val expiryDate = Date(now.time + EXPIRATION_TIME)  // 1시간
    
    return Jwts.builder()
        .setSubject(userId.toString())
        .claim("userId", userId)
        .setIssuedAt(now)
        .setExpiration(expiryDate)
        .signWith(SignatureAlgorithm.HS512, SECRET_KEY)
        .compact()
}

// JWT 토큰 검증
fun validateToken(token: String): Boolean {
    try {
        Jwts.parser()
            .setSigningKey(SECRET_KEY)
            .parseClaimsJws(token)
        return true
    } catch (e: ExpiredJwtException) {
        return false
    } catch (e: UnsupportedJwtException) {
        return false
    }
}

// 사용자 ID 추출
fun getUserIdFromToken(token: String): Long {
    val claims = Jwts.parser()
        .setSigningKey(SECRET_KEY)
        .parseClaimsJws(token)
        .body
    return claims["userId"].toString().toLong()
}
```

### 2.4 보안 고려사항

#### ❌ 보안 취약점과 ✅ 해결책

| 취약점 | 문제 | 해결책 |
|--------|------|--------|
| 토큰 탈취 | 공격자가 토큰 탈취 | Access Token 유효기간 짧게 (1시간) |
| 토큰 재사용 | 탈취된 토큰 계속 사용 | Refresh Token으로 자동 갱신 |
| 서명 검증 없음 | 토큰 위변조 | 모든 요청에서 서명 검증 |
| 민감 정보 저장 | 토큰에서 정보 노출 | 민감한 정보(비밀번호)는 저장 X |
| 토큰 로그아웃 | 로그아웃 후에도 유효 | 로그아웃 시 Refresh Token 제거 |

#### 실제 구현 예시

```kotlin
// 로그아웃 구현
@Transactional
fun logout(accessToken: String): String {
    val userId = jwtProvider.getUserIdFromToken(accessToken)
    
    // Refresh Token 제거 (Redis에서)
    refreshTokenRepository.deleteByUserId(userId)
    
    return "Logged out successfully"
}

// 토큰 갱신
fun refreshAccessToken(refreshToken: String): String {
    val userId = jwtProvider.getUserIdFromToken(refreshToken)
    
    // Redis에서 Refresh Token 존재 여부 확인
    val stored = refreshTokenRepository.findByUserId(userId)
        ?: throw IllegalArgumentException("Invalid refresh token")
    
    if (stored != refreshToken) {
        throw IllegalArgumentException("Token mismatch")
    }
    
    // 새로운 Access Token 발급
    return jwtProvider.generateToken(userId)
}
```

## 3️⃣ Redis 캐싱

### 3.1 왜 Redis를 선택했나?

```
CoverCloud의 요구사항:
- 자주 조회되는 데이터: 커버곡, 트렌딩, 댓글
- 빠른 응답: API 응답 < 200ms
- 세션 관리: 사용자 토큰

Redis의 장점:
✅ In-memory 저장으로 매우 빠름 (마이크로초 단위)
✅ TTL 지원으로 자동 만료
✅ Pub/Sub로 실시간 통신 가능 (향후)
✅ Spring Integration 잘 지원
```

### 3.2 캐싱 전략

#### 1️⃣ 캐시-어사이드 (Cache-Aside) 패턴
```kotlin
// 가장 일반적인 패턴: 읽기
fun getCover(coverId: Long): CoverResponse {
    // 1단계: 캐시에서 조회
    val cached = redisTemplate.opsForValue().get("cover:${coverId}")
    if (cached != null) {
        return cached as CoverResponse
    }
    
    // 2단계: DB에서 조회
    val cover = coverRepository.findByIdOrNull(coverId)
        ?: throw NotFoundException("Cover not found")
    
    // 3단계: 캐시에 저장
    redisTemplate.opsForValue().set(
        "cover:${coverId}",
        cover.toResponse(),
        Duration.ofHours(1)  // 1시간 TTL
    )
    
    return cover.toResponse()
}

// @Cacheable로 간단하게
@Cacheable(value = "covers", key = "#coverId")
fun getCover(coverId: Long): CoverResponse {
    // 로직은 동일
}
```

#### 2️⃣ 쓰기-스루 (Write-Through) 패턴
```kotlin
// 데이터 수정 시 캐시도 함께 갱신
@Transactional
fun updateCover(coverId: Long, request: CoverRequest): CoverResponse {
    // 1단계: DB 수정
    val cover = coverRepository.findByIdOrNull(coverId)
        ?: throw NotFoundException("Cover not found")
    cover.apply {
        coverTitle = request.title
        coverArtist = request.artist
    }
    coverRepository.save(cover)
    
    // 2단계: 캐시 갱신
    redisTemplate.opsForValue().set(
        "cover:${coverId}",
        cover.toResponse(),
        Duration.ofHours(1)
    )
    
    return cover.toResponse()
}

// @CachePut으로 간단하게
@CachePut(value = "covers", key = "#coverId")
fun updateCover(coverId: Long, request: CoverRequest): CoverResponse {
    // 로직
}
```

#### 3️⃣ 삭제 시 캐시 무효화
```kotlin
@Transactional
fun deleteCover(coverId: Long) {
    coverRepository.deleteById(coverId)
    
    // 캐시 삭제
    redisTemplate.delete("cover:${coverId}")
}

// @CacheEvict로 간단하게

}
```

### 3.3 캐시 키 설계 전략

```
좋은 캐시 키 설계:

✅ 명확한 구분
cover:123           # 커버곡 ID 123
user:456            # 사용자 ID 456
trending:DAILY      # 일간 트렌딩

❌ 나쁜 설계
data:123            # 어떤 데이터인지 불명확
temp                # 복수 데이터가 섞일 위험

✅ 복합 키
comment:cover:123:1     # 커버곡 123의 댓글 1
like:user:456:cover:123 # 사용자 456의 커버곡 123 좋아요
```

### 3.4 캐시 스탬피드 방지

```
문제: 인기 있는 데이터의 TTL 만료 시
동시에 많은 요청이 DB에 접근하는 문제

해결책:

1. 캐시 워밍 (Cache Warming)
   - 주기적으로 자주 사용되는 데이터 미리 로드
   
2. 스마트 TTL
   - 중요한 데이터는 TTL을 길게 설정
   
3. 자동 갱신
   - TTL 90% 도달 시 백그라운드에서 갱신

@Scheduled(fixedRate = 3600000)  // 1시간마다
fun refreshTrendingCache() {
    val trending = coverRepository.findTrending()
    redisTemplate.opsForValue().set(
        "trending:DAILY",
        trending,
        Duration.ofHours(1)
    )
}
```

### 4.1 N+1 문제

#### ❌ 문제 상황
```kotlin
// 문제: 커버곡 100개 조회 시 쿼리 101개!
val covers = coverRepository.findAll()  // 1개 쿼리
covers.forEach { cover ->
    println(cover.tags)  // 각각 1개 쿼리씩 × 100 = 100개 쿼리
}
// 총 101개 쿼리!
```

#### ✅ 해결책 1: Fetch Join
```kotlin
@Query("""
    SELECT DISTINCT c FROM Cover c
    LEFT JOIN FETCH c.tags
    WHERE c.id IN :coverIds
""")
fun findCoversByIdsWithTags(@Param("coverIds") coverIds: List<Long>): List<Cover>

// 사용
val covers = coverRepository.findCoversByIdsWithTags(coverIds)
covers.forEach { cover ->
    println(cover.tags)  // 추가 쿼리 없음!
}
```

#### ✅ 해결책 2: @EntityGraph
```kotlin
@EntityGraph(attributePaths = ["tags", "likes"])
@Query("SELECT c FROM Cover c WHERE c.id IN :coverIds")
fun findCoversByIds(@Param("coverIds") coverIds: List<Long>): List<Cover>
```

#### ✅ 해결책 3: BatchSize
```kotlin
@Entity
```

### 5.1 Feign Client 기본 사용법

```kotlin
// 1. Interface 정의
@FeignClient(
    name = "user-service",

        return UserProfileDto(id = id, nickname = "Unknown")
    }
    
    override fun getUsersByIds(ids: List<Long>): List<UserProfileDto> {

    }

// 3. 서비스에서 사용
@Service
class CoverService(
    private val userClient: UserClient
) {
    fun getCoverWithUserInfo(coverId: Long): CoverWithUserResponse {
        val cover = coverRepository.findById(coverId).orElseThrow()
        val user = userClient.getUser(cover.userId)  // 자동으로 HTTP 요청

            cover = cover.toResponse(),
```

### 7.1 JWT 검증 필터

```kotlin
@Component
class JwtAuthenticationManager(
    private val jwtProvider: JwtProvider
) {
    fun authenticate(request: ServerWebExchange): Long {
        // 1. Authorization 헤더 추출
        val token = extractToken(request)
            ?: throw IllegalArgumentException("No token provided")
        
        // 2. 토큰 검증
        if (!jwtProvider.validateToken(token)) {
            throw IllegalArgumentException("Invalid token")
        }
        
        // 3. 사용자 ID 추출
        return jwtProvider.getUserIdFromToken(token)
    }
    
    private fun extractToken(request: ServerWebExchange): String? {
        return request.request.headers.getFirst("Authorization")
            ?.substringAfter("Bearer ")
    }
}

// 쿠키에서 추출하는 경우
@Component
class CookieToAuthorizationFilter : WebFilter {
    override fun filter(exchange: ServerWebExchange, chain: WebFilterChain): Mono<Void> {
        val cookies = exchange.request.cookies
        val accessToken = cookies["accessToken"]?.firstOrNull()?.value
        
        return if (accessToken != null) {
            // 쿠키의 토큰을 Authorization 헤더로 변환
            val newRequest = exchange.request.mutate()
                .header("Authorization", "Bearer $accessToken")
                .build()
            
            val newExchange = exchange.mutate().request(newRequest).build()
            chain.filter(newExchange)
        } else {
            chain.filter(exchange)
        }
    }
}
```

## 🎓 면접에서 자주 나오는 심화 질문

### Q1. "트렌딩을 실시간으로 계산하면 성능이 저하되지 않나요?"
**A.** 맞습니다. 그래서 다음과 같이 최적화했습니다:
- 트렌딩은 Redis에 1시간 캐싱
- 매시간 배경 작업으로 미리 계산
- 캐시 miss 시에만 DB 쿼리

### Q2. "좋아요 기능에서 race condition이 발생할 수 있나요?"
**A.** 좋은 질문입니다. 예방 방법:
- 데이터베이스 UNIQUE 제약 (user_id, cover_id)
- 낙관적 락(Optimistic Locking)으로 충돌 감지
- Redis 원자성 연산 (incr)으로 좋아요 수 업데이트

### Q3. "100만 건의 사용자 데이터를 조회하면?"
**A.** 페이징으로 해결합니다:
- 한 번에 최대 100개만 조회
- API에서 페이지 번호, 크기 제한
- DB 인덱스 활용으로 빠른 조회
- 필요시 데이터베이스 샤딩

### Q4. "서비스 간 순환 의존성이 생기면?"
**A.** 아키텍처 설계로 예방:
- 각 서비스는 독립적 도메인 모델
- 필요시 별도 "중앙 서비스" 도입 (User Info Service)
- 이벤트 기반 아키텍처로 변경 (Kafka)

### Q5. "토큰 갱신 시 동시에 여러 요청이 오면?"
**A.** 다음과 같이 처리합니다:
```kotlin
@Transactional
fun refreshToken(refreshToken: String): String {
    val userId = extract(refreshToken)
    
    // 분산 락(Redis 기반) 사용
    val lock = redisTemplate.opsForValue()
        .setIfAbsent("refresh_lock:${userId}", "locked", Duration.ofSeconds(5))
    
    require(lock!!) { "Another refresh in progress" }
    
    try {
        return jwtProvider.generateToken(userId)
    } finally {
        redisTemplate.delete("refresh_lock:${userId}")
    }
}
```
