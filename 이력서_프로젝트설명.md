# CoverCloud 프로젝트 - 이력서 작성 가이드

## 📌 프로젝트 개요

### 프로젝트명
**CoverCloud** - 음악 커버곡 공유 플랫폼 백엔드 서버

### 프로젝트 규모
- **팀 규모**: 개인 또는 팀 프로젝트 (본인 역할 명시)
- **개발 기간**: [시작일] ~ [현재/완료일]
- **구조**: MSA(Microservices Architecture) 기반 5개 독립 모듈
- **코드량**: ~3,000+ 라인의 비즈니스 로직

### 프로젝트 설명 (한 문단)
> 음악 커버곡을 공유하고 검색할 수 있는 플랫폼의 백엔드 시스템입니다. 마이크로서비스 아키텍처로 설계되어 각 서비스(사용자, 커버곡, 음악)가 독립적으로 확장 가능하며, API Gateway를 통해 통합 관리됩니다. Redis를 활용한 캐싱으로 성능을 최적화하고, JWT 기반 인증과 Feign Client를 통한 서비스 간 통신을 구현했습니다.

- 공통 라이브러리를 통한 코드 중복 제거 및 재사용성 향상

#### 2. **커버곡 서비스 개발 (Cover Service)**
- 커버곡 CRUD 기능 구현 (생성, 조회, 수정, 삭제)
- 태그 기반 분류 시스템 설계 및 구현

- JPA를 활용한 복잡한 쿼리 최적화

#### 3. **사용자 관리 서비스 개발 (User Service)**
- OAuth 기반 소셜 로그인 통합 (Google, Naver, Kakao 등)

- Redis를 활용한 세션 및 토큰 관리
- Google Cloud Storage 연동 (프로필 이미지 업로드)
- 닉네임 변경, 계정 삭제 등 사용자 관리 기능

- Spotify API 또는 외부 음악 API 연동
- 중복 음악 데이터 처리 로직

- 사용자 ID를 요청 헤더에 추가 (다운스트림 서비스에서 사용)

## 📋 면접 예상 질문 및 답변

### Q1. 왜 MSA를 선택했나요?
> 각 서비스가 독립적으로 개발, 배포, 확장될 수 있어서입니다. 사용자 서비스와 커버곡 서비스가 다른 언어나 기술로 작성되어야 할 경우 유연하게 대응할 수 있고, 한 서비스의 장애가 전체 시스템에 영향을 주지 않습니다.

### Q2. JWT 토큰 관리는 어떻게 하나요?
> Access Token과 Refresh Token을 분리해서 사용합니다. Access Token은 단기(1시간)로 설정하고, Refresh Token은 Redis에 저장해서 만료 시 새로운 Access Token을 발급합니다. 이렇게 하면 토큰 탈취 시 피해를 최소화할 수 있습니다.

### Q3. 캐싱 전략은 어떻게 되나요?
> 자주 조회되는 커버곡 데이터나 인기 순위 데이터는 Redis에 캐싱합니다. TTL(Time To Live)을 설정해서 일정 시간 후 자동으로 무효화되도록 하고, 데이터 업데이트 시 즉시 캐시를 갱신합니다.

### Q4. N+1 문제는 어떻게 해결했나요?
> JPA의 fetch join을 사용하거나, @EntityGraph 어노테이션으로 즉시 로딩(eager loading)을 설정했습니다. 또한 배치 사이즈를 적절히 설정해서 쿼리 횟수를 줄였습니다.

### Q5. 서비스 간 통신이 실패하면 어떻게 되나요?
> Feign Client에 Fallback을 설정해서 대체 응답을 반환하거나, 에러 로깅 후 사용자에게 적절한 에러 메시지를 반환합니다. 중요한 기능은 재시도(Retry) 로직을 추가했습니다.

### Q6. 테스트는 어느 수준까지 작성했나요?
> Service 계층의 핵심 로직에 대한 단위 테스트를 작성했습니다. Mock 객체로 의존성을 주입해서 외부 서비스 호출 없이 테스트합니다. Controller 계층의 통합 테스트도 일부 작성했습니다.

### Q7. 만약 사용자가 급증하면?
> 데이터베이스 레플리케이션으로 읽기 부하를 분산하고, Redis 클러스터로 캐싱 용량을 늘리며, 서비스별 인스턴스를 각각 스케일 아웃할 수 있습니다. 또한 CDN을 추가해서 이미지 서빙을 최적화할 수 있습니다.
